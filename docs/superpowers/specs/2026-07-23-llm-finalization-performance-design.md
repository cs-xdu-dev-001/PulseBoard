# LLM昨日校准与筛选性能设计

日期：2026-07-23

## 目标

- 按`Asia/Shanghai`固化并持续校准昨天的官方LLM用量。
- 完整数据不被后续采样、失败或缺字段结果降级覆盖。
- 月度活动只读取`llm_usage_daily`，不在请求过程中扫描原始快照。
- 切换供应商、Key和时间范围时取消失效请求，并复用短期缓存。
- 首页不加载LLM图表代码，降低首次加载体积。

## 后端方案

不新增数据库表，继续使用：

- `token_complete=true`与`data_quality=complete`表示官方Token终值可用。
- `data_quality=sampled`表示只有可信日志样本或模型拆分不完整。
- `data_quality=unavailable`表示官方接口未返回可用统计。

新增昨日采集入口，目标日期统一按`Settings.lab_timezone`计算：

- NewAPI使用目标日`00:00:00`至次日`00:00:00`的精确时间范围查询统计和日志。
- DeepSeek平台继续请求官方日桶，但只持久化目标日期。
- `openai_gateway`不参与官方日结，继续按真实代理请求实时累加。

APScheduler每小时第15分钟执行昨日校准。失败不覆盖已有每日数据，下一小时自然重试。完整结果允许更新已有完整结果，以接收供应商延迟结算；低质量结果不能覆盖高质量结果。

`llm_usage_daily`的唯一键继续保证重复执行幂等。`__total__`保存官方总请求、Token与金额；模型行允许保持`sampled`，不得反向影响官方总计。

活动接口直接查询每日表。删除接口请求过程中的自动全表重建；历史重建函数只保留给测试和显式维护使用。

## 前端方案

API客户端接受`AbortSignal`。LLM页面维护两类模块级缓存：

- 主数据：键为`range＋source`，有效期30秒。
- 活动数据：键为`year＋source`，有效期60秒。

缓存同时保存进行中的Promise，合并React StrictMode、快速切换和重复挂载产生的相同请求。缓存命中时立即显示已有数据，再由过期请求后台更新。

切换筛选或卸载页面时调用`AbortController.abort()`。旧请求即使晚返回也不能覆盖当前筛选。手动刷新清除当前键并绕过缓存。

`App.jsx`使用`React.lazy`加载LLM和Settings页面。LLM模块中的ECharts改为`echarts/core`按需注册实际使用的图表和组件，避免完整ECharts进入页面分包。

## 错误处理

- 单一来源日结失败不影响其他来源。
- 日结失败保留已有完整行；没有历史值时保留采样或不可用状态。
- `AbortError`不显示为页面错误。
- 网络或解析错误只影响当前请求，缓存中的最后成功数据继续可见。

## 验收

- 北京时间边界下，昨日查询区间准确。
- 同一目标日期重复执行结果幂等。
- 完整行不会被采样或错误结果覆盖。
- 活动接口不触发快照重建。
- 快速切换筛选时只有最终筛选结果能更新页面。
- 同一活动键的并发请求被合并，时间范围变化不刷新全年活动。
- 首页主包不再包含LLM页面和完整ECharts。
- 后端、前端全量测试与生产构建通过。

## 不在范围

- 新增告警、部署脚本、多用户权限或审计系统。
- 修改`.env`格式和供应商配置页面。
- 新增每日校准状态表或长期请求明细。
