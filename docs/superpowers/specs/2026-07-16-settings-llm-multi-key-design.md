# Settings多API Key配置设计

## 问题

当前多Key表单位于LLM看板，Settings仍只暴露原始环境变量和两个固定密钥字段。结果是配置职责分散，用户无法在配置页直观地管理同一供应商下的多个API Key。

## 方案比较

1. 继续在LLM页配置：改动最少，但监控和配置继续混在一起，不符合页面解耦目标。
2. Settings与LLM页都保留配置：入口方便，但会形成重复界面和状态同步问题。
3. Settings作为唯一配置入口：供应商和Key集中管理，LLM页只负责查看数据。采用此方案。

## 页面结构

Settings保留现有运行配置，并新增独占整行的“LLM供应商”区域：

- 供应商是一级分组，显示供应商名、供应商ID、Key数量和整体配置状态。
- 展开供应商后显示Key列表，每个Key显示展示名、保存ID、类型、Base URL和密钥配置状态。
- 每个供应商提供“添加Key”，页面同时提供“新增供应商”。
- 新增和编辑共用同一个表单，保存后重新拉取配置并立即更新列表。
- 密钥只允许覆盖写入，不回显真实内容；编辑时留空表示保留原密钥。
- LLM看板移除新增配置表单和配置状态条，只保留筛选、刷新、统计卡片和图表。

## 视觉层级

- 运行配置采用紧凑分区，LLM供应商区域占满页面宽度，避免被小卡片挤压。
- 核心正文、字段标签和Key状态不小于14px；辅助ID可使用13px，但保持足够对比度。
- 供应商行负责定位和操作，Key行负责比较，不使用小字号说明段落承载核心信息。
- 深色和浅色主题均沿用现有PulseBoard配色、边框和光效，不引入新视觉体系。
- 桌面端优先保证横向扫描，窄屏下字段和操作自然换行，不出现文字遮挡。

## 数据流

1. Settings同时请求`/api/settings`和`/api/llm/usage/config`。
2. 普通运行参数继续通过`/api/settings`保存。
3. 单个LLM Key通过`/api/llm/usage/config`保存，仍写入`.env`。
4. 保存成功后重新请求配置列表，不在前端假设写入结果。

现有`provider_id/provider_name/source_id`模型已能表达多Key，不新增数据库表和Alembic迁移。

## 错误与验证

- 表单明确校验供应商ID和Key ID，失败时保留输入并显示后端错误。
- 前端增加可测试的分组和表单状态逻辑，先写失败测试再实现。
- 后端现有多Key写入测试继续作为回归保障。
- 完成后运行后端测试、前端测试与生产构建，并在桌面和375px宽度检查Settings页面。
